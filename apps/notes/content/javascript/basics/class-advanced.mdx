学会 `class` 的基础语法后，下一步不是继续堆更多写法，而是理解它背后的几个边界：方法放在哪里，`this` 为什么会丢，子类初始化为什么必须先 `super()`，以及什么时候不该继续继承。

这些问题会直接影响日常代码。比如一个类在页面事件里失效，一个子类构造函数报错，或者项目里出现很深的继承链，都不是语法背不熟，而是对象模型没有想清楚。

## class 是原型写法的语法

`class` 看起来像全新的对象系统，但它仍然建立在 JavaScript 的原型机制上。

```js
class Course {
  constructor(title) {
    this.title = title
  }

  getTitle() {
    return this.title
  }
}

const course = new Course('JavaScript class')
```

可以把它粗略理解成下面这种原型写法：

```js
function Course(title) {
  this.title = title
}

Course.prototype.getTitle = function () {
  return this.title
}

const course = new Course('JavaScript class')
```

这不是说日常代码应该回到旧写法，而是说明两个事实：

- 实例属性通常在每个实例自己身上
- 普通方法通常在 `ClassName.prototype` 上，被所有实例共享

这个理解能解释很多现象。比如两个实例的 `title` 不同，但 `getTitle` 方法是同一个函数。

```js
const first = new Course('基础课')
const second = new Course('进阶课')

first.title === second.title // false
first.getTitle === second.getTitle // true
```

## 实例字段和原型方法的区别

类方法默认放在原型上。

```js
class Counter {
  count = 0

  add() {
    this.count += 1
  }
}
```

每个 `Counter` 实例都有自己的 `count`，但共享同一个 `add` 方法。这样通常更省内存，也符合“同类对象共享行为”的直觉。

如果把方法写成实例字段，它会在每个实例上创建一个新的函数。

```js
class Counter {
  count = 0

  add = () => {
    this.count += 1
  }
}
```

这种写法的优点是箭头函数会保留创建时的 `this`，作为回调传递时不容易丢失。代价是每个实例都有一份自己的 `add` 函数。

可以这样取舍：

- 普通业务方法：优先写成原型方法
- 经常直接作为回调传递的方法：可以考虑实例字段箭头函数
- 实例数量非常多时：更要留意每个实例都创建函数的成本

## 方法作为回调时为什么会丢 this

`this` 由调用方式决定，不由方法写在哪里决定。

```js
class Timer {
  seconds = 0

  tick() {
    this.seconds += 1
  }
}

const timer = new Timer()

timer.tick() // this 是 timer
```

直接调用时，点号左边的对象就是这次调用的接收者。问题出现在把方法取出来之后。

```js
const tick = timer.tick

tick() // this 不再是 timer
```

事件监听、定时器、数组回调里也常见这个问题。

```js
setInterval(timer.tick, 1000)
```

这行代码传进去的是函数本身，不是“带着 timer 的调用”。常见修法有三种。

第一种：在调用处包一层。

```js
setInterval(() => {
  timer.tick()
}, 1000)
```

第二种：在构造函数里绑定。

```js
class Timer {
  constructor() {
    this.seconds = 0
    this.tick = this.tick.bind(this)
  }

  tick() {
    this.seconds += 1
  }
}
```

第三种：把方法写成实例字段箭头函数。

```js
class Timer {
  seconds = 0

  tick = () => {
    this.seconds += 1
  }
}
```

如果只是偶尔传一次回调，包一层最直观。如果类的方法经常被当作回调使用，再考虑绑定或实例字段箭头函数。

## 字段初始化和 super 的顺序

类字段会在实例创建时初始化。

```js
class Course {
  title = '未命名课程'

  constructor(title) {
    this.title = title
  }
}

const course = new Course('JavaScript class')

course.title // 'JavaScript class'
```

在这个例子里，字段先给 `title` 一个默认值，构造函数再用传入参数覆盖它。

子类会多一个限制：在访问 `this` 之前必须先调用 `super()`。

```js
class IntensiveCourse extends Course {
  constructor(title, dailyGoal) {
    super(title)
    this.dailyGoal = dailyGoal
  }
}
```

`super(title)` 会先运行父类构造函数，让父类完成自己的初始化。只有父类初始化完成后，子类才能继续给当前实例添加自己的属性。

下面这种写法会报错：

```js
class IntensiveCourse extends Course {
  constructor(title, dailyGoal) {
    this.dailyGoal = dailyGoal
    super(title)
  }
}
```

原因不是 JavaScript 对顺序吹毛求疵，而是子类实例要先由父类构造逻辑建立起来，`this` 才能安全使用。

## 私有字段不会被子类继承为可访问属性

子类继承父类的公开方法，但不能直接访问父类的私有字段。

```js
class Course {
  #completedLessons = 0

  completeLesson() {
    this.#completedLessons += 1
  }

  getProgressLabel() {
    return `${this.#completedLessons} lessons`
  }
}

class IntensiveCourse extends Course {
  readProgress() {
    return this.#completedLessons
  }
}
```

`readProgress()` 会语法错误，因为 `#completedLessons` 只在声明它的类体内可见。子类如果需要读取进度，应该通过父类提供的公开方法。

```js
class IntensiveCourse extends Course {
  readProgress() {
    return this.getProgressLabel()
  }
}
```

这条边界很重要：私有字段不是“给子类用、不给外部用”的 protected。JavaScript 当前没有 `protected` 关键字。如果希望子类参与某段逻辑，可以设计一个公开方法，或者把可复用逻辑拆成普通函数。

## 继承不要太深

继承适合表达明确的“是一种”关系。

```js
class Course {}

class VideoCourse extends Course {}
```

如果 `VideoCourse` 确实是一种课程，并且它自然复用课程的核心行为，继承是合理的。

但很多时候，需求只是“带有某种能力”。比如课程可以打折、可以收藏、可以同步远程数据。这些能力未必应该层层继承。

```js
class Course {}
class DiscountCourse extends Course {}
class FavoriteDiscountCourse extends DiscountCourse {}
class RemoteFavoriteDiscountCourse extends FavoriteDiscountCourse {}
```

这种继承链很快会变得难改：上层一变，下面所有子类都受影响；某个能力想换组合方式，也要重新设计整个层级。

这时组合通常更直接。

```js
class Course {
  constructor(title, pricing) {
    this.title = title
    this.pricing = pricing
  }

  getPrice() {
    return this.pricing.getPrice()
  }
}

class DiscountPricing {
  constructor(basePrice, discount) {
    this.basePrice = basePrice
    this.discount = discount
  }

  getPrice() {
    return this.basePrice * this.discount
  }
}

const course = new Course('JavaScript class', new DiscountPricing(100, 0.8))
```

组合的思路是：对象不一定要通过继承获得能力，也可以持有另一个对象，把某个职责交给它。

## 用 class 设计对象时先问职责

不要一看到数据和函数就急着建类。先问三个问题：

- 这个对象负责维护什么状态？
- 哪些操作必须通过它的方法完成？
- 它依赖哪些外部能力？

比如一个购物车可以负责维护商品列表和计算总价。

```js
class Cart {
  #items = []

  addItem(item) {
    this.#items.push(item)
  }

  get totalPrice() {
    return this.#items.reduce((total, item) => total + item.price * item.quantity, 0)
  }
}
```

但“把购物车保存到服务器”不一定应该写进 `Cart`。保存动作依赖网络请求、错误处理和后端接口，可能更适合交给单独的服务函数。

```js
async function saveCart(cart, apiClient) {
  await apiClient.post('/cart', cart.toJSON())
}
```

类越清楚自己的职责，后续改动越容易。类承担太多事情时，继承、私有字段和 getter/setter 都救不了设计问题。

## 练习

练习 1：观察实例属性和原型方法的区别。创建两个 `Counter` 实例，比较它们的 `count` 和 `add`。

参考代码：

```js
class Counter {
  count = 0

  add() {
    this.count += 1
  }
}

const first = new Counter()
const second = new Counter()

first.count === second.count // true，因为都是数字 0
first.add === second.add // true，因为方法来自同一个原型

first.add()

first.count // 1
second.count // 0
```

练习 2：修复丢失的 `this`。下面代码会在定时器里丢失 `this`，把它改成安全写法。

```js
class Timer {
  seconds = 0

  tick() {
    this.seconds += 1
  }
}

const timer = new Timer()

setInterval(timer.tick, 1000)
```

参考代码：

```js
setInterval(() => {
  timer.tick()
}, 1000)
```

练习 3：把继承改成组合。不要创建 `DiscountCourse extends Course`，而是让 `Course` 接收一个 `pricing` 对象。

参考代码：

```js
class Course {
  constructor(title, pricing) {
    this.title = title
    this.pricing = pricing
  }

  getPrice() {
    return this.pricing.getPrice()
  }
}

class FixedPricing {
  constructor(price) {
    this.price = price
  }

  getPrice() {
    return this.price
  }
}

const course = new Course('JavaScript class', new FixedPricing(99))

course.getPrice() // 99
```
